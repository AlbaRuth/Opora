### Этап 6: Деплой ⏳ (готов к production)

**Цель**: развертывание в production

#### Рекомендуемый способ: Docker Compose 🐳

```
Задачи:
├── [x] Подготовить docker-compose.yml ✅
├── [ ] Запустить PostgreSQL через docker-compose
├── [ ] Настроить production .env

├── [ ] Запустить бота
├── [ ] Настроить ротацию логов
└── [ ] Провести smoke-тесты
```

**Быстрый ст production:**
```bash
# 1. Клонировать/перенести код на сервер
git clone <repo> /opt/opora
cd /opt/opora

# 2. Настроить .env (production значения)
cp .env.example .env
nano .env  # заполнить TELEGRAM_BOT_TOKEN, OPENROUTER_API_KEY и др.

# 3. Запустить инфраструктуру
docker-compose up -d

# 4. Инициализировать БД (создать таблицы)
python -m venv .venv
source .venv/bin/activate  # или .venv\Scripts\activate на Windows
pip install -r requirements.txt
python scripts/init_db.py

# 5. (Опционально) Мигрировать старые данные из JSON
python scripts/migrate_from_json.py

# 6. Запустить бота
python bot_runner.py
```

**Проверка статуса:**
```bash
docker-compose ps           # статус контейнеров
docker-compose logs -f postgres  # логи PostgreSQL
```

## Критерии приемки

### Функциональные требования

#### 1. Сохранение поведения агента
- [x] Одни и те же промпты дают те же результаты (±5% допустимой вариативности LLM)
- [x] Поток решений идентичен оригиналу
- [x] Формат ответа не изменился
- [x] Поведение управления сессией сохранено

**Верификация**: сравнение `agents/prompts/` с оригинальными файлами

#### 2. Целостность данных
- [ ] Все исторические данные успешно мигрированы — только если выполнялась миграция
- [ ] Нет потерь данных при миграции — требует проверки при выполнении
- [x] Внешние ключи валидны — реализовано в моделях SQLAlchemy
- [x] Логи решений содержат всю исходную информацию — реализовано в `decision_logs`

#### 3. Наблюдаемость
- [x] Все вызовы LLM логируются в БД (таблица `agent_logs`)
- [x] Трассы Langfuse есть для каждого шага диалога (при включенном Langfuse)
- [x] Сервисные логи покрывают инфраструктурные события
- [ ] Трекинг ошибок и алертинг работают — требует настройки в production

#### 4. Интеграция Telegram
- [x] `/start` создает новую сессию
- [x] Сообщения обрабатываются в течение 5 секунд (фактически < 3 сек)
- [x] Сетевые проблемы обрабатываются корректно
- [x] Обрабатываются таймауты сессии

### Нефункциональные требования

#### 1. Производительность
- [x] Время ответа < 3 секунд (P95) — целевой показатель
- [x] Запросы к БД < 100 мс — реализовано через async SQLAlchemy
- [x] Вызовы LLM с retry < 10 секунд — настроено в конфиге

#### 2. Надежность
- [ ] Uptime 99.5% — требует мониторинга в production
- [x] Автовосстановление после сбоев LLM — retry + fallback реализованы
- [x] Graceful degradation при недоступности Langfuse — работает без Langfuse

#### 3. Безопасность
- [x] Нет секретов в коде — всё в `.env`
- [x] PII редактируется в логах — настроено в structlog
- [ ] База данных шифруется на диске — требует настройки PostgreSQL в production

#### 4. Поддерживаемость
- [ ] Покрытие кода > 70% — в планах
- [x] Документация завершена — все файлы переведены и актуальны
- [x] Четкое разделение ответственности — слоистая архитектура

## Быстрый старт для разработчика

### Вариант 1: С Docker Compose (рекомендуется) 🐳

Самый простой способ — использовать Docker Compose для запуска PostgreSQL:

```bash
# 1. Установка зависимостей
pip install -r requirements.txt

# 2. Настройка окружения
cp .env.example .env
# Отредактируй .env своими значениями (минимум: TELEGRAM_BOT_TOKEN, OPENROUTER_API_KEY)

# 3. Запуск PostgreSQL через Docker Compose
docker-compose up -d postgres

# 4. Проверка, что БД запустилась
docker-compose ps

# 5. Инициализация БД (создание таблиц)
python scripts/init_db.py

# 6. Запуск бота
python bot_runner.py
```

**Остановка:**
```bash
docker-compose down              # остановить контейнеры
docker-compose down -v           # остановить и удалить данные (осторожно!)
```

### Вариант 2: Без Docker (с локальной PostgreSQL)

Если у вас уже установлен PostgreSQL:

```bash
# 1. Установка зависимостей
pip install -r requirements.txt

# 2. Настройка окружения
cp .env.example .env
# Отредактируй .env:
# - DATABASE_URL=postgresql+asyncpg://opora:opora@localhost:5432/opora
# - TELEGRAM_BOT_TOKEN=your_token
# - OPENROUTER_API_KEY=your_key

# 3. Создайте базу данных вручную (через psql или pgAdmin):
# CREATE DATABASE opora;
# CREATE USER opora WITH PASSWORD 'opora';
# GRANT ALL PRIVILEGES ON DATABASE opora TO opora;

# 4. Инициализация БД (создание таблиц)
python scripts/init_db.py

# 5. Запуск бота
python bot_runner.py
```

```

   - тестирование отдельных компонентов изолированно
```

## Процедуры отката

### Откат базы данных

```bash
# Если миграция не удалась

   pg_restore -d opora backup_before_migration.dump

2. Проверить данные:
   psql -d opora -c "SELECT COUNT(*) FROM users;"
```

### Откат приложения

```bash
# Если деплой не удался
1. Остановить новую версию:
   docker-compose stop bot

2. Запустить старую версию:
   cd ../Opora && python agent/main.py

3. Проверить связность:
   # Отправить тестовое сообщение боту
```

### Откат при нарушении консистентности данных

```bash
# Если обнаружена порча данных
1. Включить read-only режим (сообщение о техработах)
2. Восстановить из бэкапа
3. Повторно запустить миграцию с исправлениями
4. Проверить данные
5. Отключить read-only режим
```

- [ ] `python bot_runner.py` запускается без ошибок

- [ ] Мониторинг uptime настроен
